# Style Guide

## Design Principles

- Clean and personal
- Lightweight
- Responsive
- Accessible
- Image-friendly
- Easy to maintain

## Content Tone

The writing should feel personal, clear, thoughtful, and specific.

Avoid:

- generic marketing language
- filler phrases
- overly robotic summaries
- unnecessary jargon

Prefer:

- concrete observations
- short summaries
- useful titles
- personal but concise explanations

## HTML Style

- Use semantic HTML.
- Keep markup readable.
- Preserve existing Bootstrap/template classes when possible.
- Keep one clear `h1` per page.

## Nested Detail Page Template

All section detail pages (cinema, tech-blog, photography, algorithmic-art) share one layout, modeled on `cinema/tar.html`:

```html
<main id="main-collapse">
  <div class="row">
    <div class="col-xs-12 col-md-8">

      <div class="section-container-spacer">
        <h1>Title</h1>
        <p class="published-date">Published: Month Year</p>
        <p>Intro paragraph.</p>
      </div>

      <div class="section-container-spacer">
        <!-- hero block: <img class="img-responsive"> for cinema/photography,
             or a <div class="viz-page"> with canvas + .info-panel for algorithmic-art -->
      </div>

      <p>Follow-up paragraph.</p>
      <p>Follow-up paragraph.</p>
    </div>
  </div>
</main>
```

Notes:

- Every nested page uses `<header id="site-header">` so the sidebar/mobile bar are generated by `sidebar.js`.
- Use `published-date` for the human-readable date line. The machine-readable date stays in `metadata.json` (`YYYY-MM-DD`).
- For algorithmic-art pages, the visualization canvas and its `About This Visualization` info panel sit side-by-side inside a `.viz-page` grid (defined in `assets/css/algorithmic-art.css`) and stack vertically below 900px.

## Image Style

- Use optimized images in listing grids.
- Use original/high-quality images only when the page needs them.
- Always include useful alt text for meaningful images.

## Metadata Style

- Titles should be concise.
- Summaries should work as card descriptions.
- Tags should be consistent and reusable.
- Dates should use `YYYY-MM-DD`.

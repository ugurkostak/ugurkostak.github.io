Ugur Kostak

Accessibility

Published: January 2024

A short note on why accessibility belongs at the start of any web project rather than at the end. This piece collects a few habits that have made the biggest difference in my own work.

Start with the markup

Most accessibility wins come from semantic HTML—headings in order, buttons used as buttons, labels tied to inputs. Frameworks and ARIA can refine the result, but they cannot substitute for a clean document outline.

Test as you build

Run a keyboard-only pass on every page. Switch on a screen reader once a week. Use Lighthouse and axe early, not as a release gate. Small, frequent checks catch problems before they become structural.

Make it a habit

Accessibility is less a checklist than a default posture: write the simplest markup that expresses the meaning, prefer real text to images of text, never rely on color alone. Done consistently, it stops feeling like extra work.
